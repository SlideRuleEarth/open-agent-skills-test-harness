# Design: MCP Server Support in the Agent-Skills Test Harness

**Status: draft for discussion** ¬∑ CLI versions verified against: claude 2.1.113 ¬∑ codex 0.140.0 ¬∑ copilot 1.0.64 ¬∑ agy 1.1.1

## 1. Context and goals

We are developing a custom MCP server (SlideRule data access). The harness needs to:

1. **Let scenarios exercise MCP servers** ‚Äî declare a server (local stdio process or remote HTTP/SSE endpoint), have the agent under test call its tools, and assert on that usage.
2. **Control access** ‚Äî MCP stays hermetically OFF by default; only scenarios that explicitly declare servers get them, optionally restricted to specific tools. Credentials (API keys, bearer tokens) come from environment variables at run time and never appear in scenario YAML or archived artifacts.

All four adapters (claude, codex, copilot, antigravity) are in scope, with a phased rollout grounded in what each CLI can actually do.

### 1.1 Pre-Phase-0 state ‚Äî a hermeticity bug (historical; fixed below)

MCP was intentionally disabled from the start, but before Phase 0 only the Claude adapter actually enforced it (line references are to the pre-Phase-0 code):

- **claude** ‚Äî was already hermetic. `--strict-mcp-config` sat in `_HERMETIC` (`adapters/claude.py:63-68`) with no `--mcp-config`, so no servers loaded.
- **codex** ‚Äî **leaked**. The isolated HOME symlinked `~/.codex` wholesale (`isolation.py:130-149`), so any `[mcp_servers.*]` in the user's real `config.toml` loaded in every run.
- **copilot** ‚Äî **leaked**. `--disable-builtin-mcps` (`adapters/copilot.py:73`) only disabled the bundled GitHub server; the user's real `~/.copilot/mcp-config.json` still loaded through the symlinked HOME.
- **antigravity** ‚Äî **leaked** (latently). `<iso_home>/.gemini/config/mcp_config.json` was a symlink to the real file; benign at the time only because that file happened to be empty.

Fixing this was Phase 0, worth doing regardless of the rest of this design.
**Update: Phase 0 is implemented** (revised across several review rounds ‚Äî see ¬ß8 for the mechanics). Findings from implementation that corrected this design, all verified against the installed CLIs: codex's `-c mcp_servers={}` does **not** clear the persisted table (deep-merge), disabling a name codex doesn't load *breaks* the run, its plugins are a separate MCP channel, and its effective server set depends on the run's **cwd and env** (trusted-project `.codex/config.toml`, `$CODEX_HOME`) ‚Äî so enumeration must run in the child's exact context; copilot/agy each have additional channels beyond the single user config file (plugins ‚Äî including copilot's `config.json` `cache_path` records and agy's `plugins.json` ‚Äî workspace configs across agy's *four* customization roots, `COPILOT_HOME`, and copilot's Windows ODR registry, which is enumerable), and copilot rejects `{}` as an MCP config (¬ß2).

## 2. Per-CLI MCP capability survey

Everything marked **(verified)** was checked against the installed CLI's help output or embedded docs; **(inferred)** items are documented behavior not confirmable from `--help` and carry a verification task (¬ß9).

### claude 2.1.113
- `--mcp-config <configs...>` ‚Äî loads servers from JSON files or inline strings (verified). Paired with the already-passed `--strict-mcp-config` ("only use MCP servers from --mcp-config"), declared servers become the *only* servers ‚Äî hermetic opt-in for free.
- Tool gating: `--allowedTools` / `--disallowedTools`; MCP tools are named `mcp__<server>__<tool>`; `mcp__<server>` allows the whole server (flags verified; naming inferred).
- Transports: stdio (`command`/`args`/`env`) and remote (`"type": "http"|"sse"`, `url`, `headers`) in the `mcpServers` JSON (inferred).
- Caveat: the harness runs with `--dangerously-skip-permissions` (`adapters/claude.py:100-101`); `--allowedTools` is a *permission* allowlist and is likely moot under it (inferred ‚Äî see ¬ß6 decision C2).

### codex 0.140.0
- No MCP flag on `exec`; the mechanism is `-c key=value` dotted-path TOML overrides, valid on `exec` (verified).
- **`-c` deep-merges with the persisted config ‚Äî it cannot clear a table** (verified 0.140.0: with `[mcp_servers.review_fixture]` in config.toml, `-c mcp_servers={}` leaves the server enabled in `codex mcp list --json`). Per-server `-c mcp_servers.<name>.enabled=false` *does* disable it (verified) ‚Äî so hermetic-off requires enumerating configured names, and Phase 1's "declared servers only" likewise means disabling every user server not in the declared set.
- **Enumeration must match codex's effective view exactly**: a `-c ‚Ä¶enabled=false` for a name codex does NOT load creates an incomplete top-level entry and the run dies with "invalid transport" (verified). The **only** enumerator is codex itself (`codex --disable plugins mcp list --json`) ‚Äî there is no offline config parse: a hand-parsed subset of one config.toml misses codex's system/managed layers, and a later successful main invocation could then launch an un-disabled server, so any failure of the CLI enumeration **fails closed**. Profiles don't contribute: the legacy `profile = "x"` config key is rejected outright by 0.140.0 ("no longer supported; use `--profile`"), the harness never passes `--profile`, and inactive `[profiles.*]` tables are ignored (all verified).
- **The effective view depends on the run's cwd and env** (verified 0.140.0): a *trusted* project contributes its `<git root>/.codex/config.toml` servers (untrusted projects don't; the root is resolved via git, so a subdir cwd inside a trusted repo still loads it), and `$CODEX_HOME` moves the global config. So enumeration runs `mcp list` with the **child's exact cwd and env** (exec.execute() computes the subprocess env *before* argv and hands it to the adapter via `RunOptions.effective_env`; probes get their own fresh workspace + env). Its JSON (an array of objects with a string `name`) is validated strictly ‚Äî `null`, `{"servers": [‚Ä¶]}` or any other drifted shape **fails closed** rather than reading as "no servers". Results are **not cached**: the effective view depends on the full execution context (resolved binary/PATH, git-root state, system/managed config), more than any home+cwd+mtime key could capture, so a stale entry could disable the wrong set.
- **Managed/MDM config can outrank the kill-switch, so the overrides are post-verified**: a higher-precedence configuration layer can override a `-c ‚Ä¶enabled=false` (and `--disable plugins`). After building the disable overrides the adapter re-runs codex's own enumeration **with the overrides applied**, in the same cwd/env, and confirms every server is reported disabled (an entry counts as enabled unless it carries `"enabled": false`). Any server still enabled ‚Äî or any failure to obtain that post-override view ‚Äî **fails closed**.
- **Plugins are an MCP channel of their own**: a codex plugin can ship `.mcp.json` servers that never appear in config.toml, and disabling them by name would hit the incomplete-entry failure above. `plugins` is a stable feature flag; `--disable plugins` (valid at top level, verified) removes the channel wholesale ‚Äî plugin-provided skills go with it.
- Server names are restricted to `[A-Za-z0-9_-]` by `codex mcp add` (verified), i.e. always TOML-bare-key-safe in a `-c` dotted path. The `-c` parser cannot address *quoted* key segments (verified: it errors with "failed to load configuration"), so a hand-edited exotic name fails closed rather than loading.
- **`RunOptions.extra_args` are a config channel of their own**: appended after the kill-switch overrides, a `-c`/`--config` token there outranks them (verified 0.140.0: the later of two duplicate `-c mcp_servers.<n>.enabled=` overrides wins), and `--profile`/`-p`, `--cd`/`-C`, or `--enable` switches which configuration codex resolves ‚Äî all after the post-verify has run. `build_argv` fails closed on every spelling of those flags (standard runner cells never populate `extra_args`; only the programmatic API can).
- No feature-level kill-switch: `codex features list` has no `mcp` entry (verified).
- Config vocabulary (verified via `codex mcp add --help` and binary strings): stdio `command`/`args`/`env`; remote `url` (streamable HTTP) with `bearer_token_env_var`, `http_headers`, `env_http_headers` (header value taken from a named env var); `enabled_tools`/`disabled_tools`; `startup_timeout_sec`/`tool_timeout_sec`.
- `bearer_token_env_var` is notable: the secret never appears in argv or on disk ‚Äî only the env-var *name* does.
- Gap: no legacy SSE support surfaced anywhere (inferred gap).
- The harness parser already normalizes `mcp_tool_call` items into TOOL_CALL events (`adapters/codex.py:168-195`).

### copilot 1.0.64
- `--additional-mcp-config <json>` ‚Äî JSON string or `@file`, repeatable, augments `~/.copilot/mcp-config.json` for the session (verified).
- Per-server vocabulary via `copilot mcp add`: `--transport stdio|http|sse` (all three), `--env`, `--header`, `--timeout`, and per-server `--tools "*"|list|""` filter (verified).
- Also useful: `--secret-env-vars <names>` redacts those env values from output (verified); `COPILOT_HOME` overrides `~/.copilot` (verified); `--disable-mcp-server <name>`, `--allow-tool`/`--deny-tool` with `MyMCP(tool)` syntax (verified).
- No "ignore user mcp-config" flag exists ‚Äî hermeticity must come from masking the file in the isolated HOME.
- **Server sources beyond `~/.copilot/mcp-config.json`** (verified in the 1.0.64 bundle): workspace `.mcp.json` / `.github/mcp.json` (and `.vscode/mcp.json`, `"servers"` key) discovered by walking *up* from the cwd; and installed plugins, whose definitions can declare `mcpServers`. All of them must be neutralized for hermetic-off, and `COPILOT_HOME` redirects the home-based ones wholesale.
- **Plugin loading doesn't go through `installed-plugins/` alone**: plugin records live in `config.json` (`installedPlugins`, each with an absolute `cache_path` the loader follows even when `installed-plugins/` is empty ‚Äî verified live). `config.json` also holds auth tokens and is JSONC (comment lines), so it must be *sanitized* (drop `installedPlugins`/`enabledPlugins`, keep the rest), not replaced.
- **The empty mcp-config shape is `{"mcpServers": {}}`** ‚Äî a bare `{}` fails validation with "mcpServers: Required" (verified 1.0.64) and kills the session before execution.
- **Windows ODR registry servers are enumerable** (1.0.64 bundle, "ODR load/convert" pipeline): the registry value `Command` under `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Mcp` names a command line; copilot executes `<that command> mcp list`, parses the JSON `servers` array, and registers each named server after sanitizing the name to `[0-9a-zA-Z_./@-]` (others ‚Üí `_`) and de-colliding with an FNV-1a-32 suffix of the original name. No flag/env/setting disables the mechanism, but the registered servers land in the same named map `--disable-mcp-server` filters ‚Äî so the harness reproduces the exact resolution (hash verified byte-identical against the JS algorithm) and disables every resolved name; over-disabling is safe (`--disable-mcp-server` tolerates unknown names, verified). Any failure to positively enumerate while the registry gate is on **fails closed** (the run/probe fails rather than executing with registry servers live).

### antigravity (agy) 1.1.1
- No MCP flags at all (verified absence). Discovery is purely file-based: `~/.gemini/config/mcp_config.json` ‚Äî `{"mcpServers": {name: {command, args, env}}}` for stdio, `serverUrl` (SSE) or `url` for remote (verified via embedded docs + changelog).
- Plugins are a second file-based channel: "MCP Servers defined in `plugins/<name>/mcp_config.json`" (verified via embedded plugin docs) ‚Äî hermetic-off must mask that file inside every installed plugin too. `~/.gemini/config/plugins.json` can additionally register *external* plugin directories by absolute path (their MCP configs live outside every masked tree), so that registration file is masked as well.
- The **run workspace** is a third channel (verified 1.1.1 with sentinel stdio servers launched at session start): `--add-dir` exposes it, and agy recognizes **four** customization roots ‚Äî `.agents/`, `.agent/`, `_agents/`, `_agent/` ‚Äî loading, per root, `mcp_config.json`, every `plugins/<name>/mcp_config.json`, and `plugins.json` (the entries/inherits registration schema from the embedded json_configs.md ‚Äî an *external* plugin dir registered from a workspace `plugins.json` had its MCP server launched, verified). Ancestors of the workspace are NOT scanned (verified). A seeded scenario file could re-enable MCP, so the runner neutralizes all sixteen patterns after seeding (`workspace_config_masks` ‚Äî per root: `mcp_config.json`, `plugins/*/mcp_config.json` plus its dot-inclusive companion `plugins/.*/mcp_config.json` (glob's `*` skips hidden names, and agy loads dot-prefixed plugin dirs too, verified), and `plugins.json`), and model probes anchor `--add-dir` to a fresh private 0700 temp dir instead of the shared system temp root (whose planted `.agents/` would load ‚Äî `/tmp` is world-writable on Linux).
- **No tool-allowlist mechanism** (verified absence) ‚Äî documented gap.
- Transcript tool names look like `mcp_<server>_<tool>` with single underscores (inferred from binary strings).

## 3. Scenario schema extension

New optional top-level field on evals and scenarios. Absent ‚áí MCP hermetically off (the default today, post-Phase-0 for all adapters).

```yaml
mcp_servers:                              # dict[str, server]; key = server name = tool namespace
  sliderule:                              # ‚Üí tools surface as mcp__sliderule__<tool>
    command: python3                      # stdio transport (XOR with url); bare names stay PATH-resolved
    args: [fixtures/sliderule_mcp.py]     # entries naming files under the scenario dir are absolutized
    env:
      SLIDERULE_DOMAIN: slideruleearth.io
      SLIDERULE_API_KEY: ${SLIDERULE_MCP_TOKEN}    # interpolated from harness process env
    tools: [atl06p, atl03x]               # optional allowlist; omit ‚áí all tools

  sliderule-remote:
    url: https://sliderule.example.com/mcp     # remote transport (XOR with command)
    transport: sse                        # http (default) | sse
    headers:
      Authorization: Bearer ${SLIDERULE_MCP_TOKEN}
```

Validation rules (added to `validate_spec`, `spec.py:237-395`, so misconfiguration fails before any tokens are spent):
- Exactly one of `command` / `url` per server; unknown keys ‚Üí error.
- `${VAR}` interpolation in `env` values, `headers` values, and `url`, resolved from the harness process environment at load time; unresolvable ‚Üí error naming the variable.
- Every interpolated value is registered in a per-run **secrets set** (used for artifact redaction, ¬ß5.3).
- Path absolutization is selective, resolving against the spec's `base_dir()` (same convention as `files:`, `spec.py:159-182`) ‚Äî the agent runs in a tempdir workspace, so scenario-relative paths would otherwise break. `command` is absolutized only when it contains a path separator *and* names an existing file under `base_dir()`; a bare executable name (`python3`) is left for PATH lookup. Each `args` entry is absolutized only when it names an existing file or directory under `base_dir()`; flags and other non-path arguments pass through verbatim.
- `tools:` on an antigravity target ‚Üí warning ("advisory only"); `mcp_servers` + `isolated: false` on an antigravity target ‚Üí error (¬ß4, agy).

### Plumbing (mirrors the existing `output_schema` / `env` pattern)

1. `EvalSpec.mcp_servers: dict` ‚Äî parsed in `_spec_from_raw` (`spec.py:433-500`).
2. `RunOptions.mcp_servers: Optional[dict]` and `RunOptions.mcp_scratch_dir: Optional[str]` (`adapters/base.py:27-45`).
3. Runner populates both per cell (`runner.py:345-352`). The scratch dir is a per-cell `ase-mcp-` tempdir created alongside the isolated home and deleted in the same `finally` (`runner.py:359-361`). It is deliberately **outside the workspace**: the workspace is relocated into `artifacts/**/workspace` and inlined into `report.md` (`runner.py:413-415`, `render_report`), which would archive secret-bearing config files.
4. Each adapter's `build_argv` / `env` consumes them (¬ß4).

AND every enumerable server is disabled by name via `--disable-mcp-server` on argv ‚Äî the user config (resolved from the child's env, so a scenario's `COPILOT_HOME:` override enumerates what the child will read ‚Äî a *relative* home resolving against the child's cwd, as copilot itself does; on POSIX a set-but-empty `HOME` counts as relative, since Node's `homedir()` returns `""` for it and copilot then reads `<its cwd>/.copilot`, while an empty `COPILOT_HOME` is unset to copilot everywhere ‚Äî both verified 1.0.64. win32 home resolution mirrors Node/libuv exactly via a fully-qualified predicate (not `ntpath.isabs`, which mis-reports a bare UNC share root as relative on Python 3.10, fixed in 3.11): rooted-but-driveless homes take the child cwd's drive, a complete UNC share root ‚Äî including the extended `\\?\UNC\server\share` form, which Python 3.11+ `splitdrive` returns whole with an empty tail while 3.10 splits it as `\\?\UNC` + rooted tail, both accepted since its joins insert the separator exactly like Node (verified 3.10‚Äì3.14) ‚Äî and rooted device-namespace paths (`\\?\C:\x`) are accepted as-is, but a *drive-relative* home (bare `D:`/`C:x`, per-drive cwd unknowable and not equated to the child cwd ‚Äî `COPILOT_HOME="D:"` persists to `D:\`), a bare drive-letter device form (`\\?\C:`/`\\.\C:`, whose joins drop the separator ‚Äî `\\?\C:mcp-config.json` ‚Äî naming paths copilot never reads), and an absent or <3-char `%USERPROFILE%` (absent ‚Üí libuv falls back to `GetUserProfileDirectoryW`, the real profile dir resolved from the process token outside this env; Argv coverage extends to probes, judge runs, and non-isolated runs. Non-isolated plugin servers can't be enumerated (names live inside plugin definitions) ‚Äî documented gap; the runner warns when isolation is off.
- **antigravity** (`adapters/antigravity.py`): only injection point is the discovery file. Secrets live only in the isolated-home tempdir, deleted right after execution and never archived. Consequence: **MCP hermeticity (and later MCP injection) on agy requires isolation ON** (the default): an overlay build failure fails the cell closed, `isolated: false` gets a loud warning today and is a validation error once `mcp_servers` exists (Phase 1).

### Isolation-layer generalization (implemented, Phase 0)

`build_isolated_home` (`isolation.py`) gained a third leaf type ‚Äî a **config mask**: adapter-declared HOME-relative paths materialized instead of symlinked, as a real file with supplied content (each CLI's accepted empty shape ‚Äî copilot rejects a bare `{}`, ¬ß2), a callable that *sanitizes* the real file, or, with content `None`, an empty real directory. E.g. `isolation_config_masks = {".copilot/mcp-config.json": '{"mcpServers": {}}', ".copilot/installed-plugins": None, ".copilot/config.json": <sanitizer>}`, `{".gemini/config/mcp_config.json": '{"mcpServers": {}}'}`. `plugin_registry_config_masks` applies the same idea inside every plugin of a registry (agy's `plugins/<name>/mcp_config.json`). Mask paths are validated (relative, no `..`, no drive prefixes, not empty/dot-only). `isolation_config_homes` entries carry the HOME subdir the env var stands in for, so masks re-root into mirrored custom homes (`$COPILOT_HOME/mcp-config.json`). A shared `build_mcp_masked_home` builds a mask-only overlay (skills untouched) for model probes and judge runs, which otherwise execute against the real HOME. One mechanism serves the Phase 0 hermeticity fix and the copilot/agy injection path (Phases 2‚Äì3).

## 5. Access control

### 5.1 Opt-in (server level)
Default stays hermetic via the kill-switch column in ¬ß4. When `mcp_servers` is declared, only those servers are reachable: claude's `--strict-mcp-config`; codex's per-name `enabled=false` disables for every user server outside the declared set (a full-table `-c mcp_servers={}` override was disproven ‚Äî it deep-merges, ¬ß2) plus `--disable plugins`; copilot/agy's masked user config.

### 5.2 Tool gating (tool level)
Per-server `tools:` compiles to each CLI's native filter ‚Äî codex `enabled_tools` and copilot per-server `tools` are hard filters (verified). Claude: `--allowedTools` is likely advisory under `--dangerously-skip-permissions`; Phase 1 accepts advisory gating (assertions still catch off-limits usage), with a documented upgrade path (¬ß6-C2) if hard enforcement matters. agy: no mechanism; validation warns.

### 5.3 Credentials
- `${VAR}` interpolation from the harness process env only; fail-fast at validation; values never committed in YAML.
- Prefer CLI-native env-var **indirection** where it exists (codex `bearer_token_env_var`, `env_http_headers`) ‚Äî the value then never touches disk or argv at all.
- Scratch config files created `0600`, outside the workspace, deleted post-run.
- **Redaction pass**: every interpolated secret value is scrubbed from `stdout.jsonl`, `stderr.txt`, `events.json`, `result.json` (including recorded `argv`), and `report.md` in `_write_artifacts` / `render_report` (`runner.py:488-495`, `runner.py:766`). This also covers the case no CLI flag can: a tool *result* echoing a token back into the transcript.

## 6. Decision points

**A. Config delivery for claude/copilot: inline argv JSON vs scratch file vs workspace `.mcp.json`.**
Inline JSON leaks resolved secrets into `result.json` (argv is archived) and needs shell-safe quoting. Workspace files are archived and inlined into `report.md`; claude ignores workspace `.mcp.json` under `--strict-mcp-config` anyway. ‚Üí **Scratch file** in the per-cell tempdir, deleted post-run.

**B. Codex delivery: `-c` overrides vs materialized `$CODEX_HOME/config.toml`.**
`-c` needs no isolation surgery, keeps HTTP secrets env-indirect natively, and its one exposure (stdio env literals in argv) is covered by redaction. A materialized config home means converting `.codex` from a wholesale symlink into a masked dir with a merged TOML (preserving auth) ‚Äî heavier and riskier. ‚Üí **`-c` overrides**, with config-home materialization as the fallback if TOML array/inline-table `-c` values prove unreliable (verification task).

**C1. Allowlist shape: flat top-level `allowed_tools: [mcp__sliderule__*]` vs per-server `tools:`.**
Flat matches claude's flag but forces claude's naming onto codex/copilot (whose native filters take bare tool names per server) and repeats server names. ‚Üí **Per-server `tools:`**; adapters compile to their native form. A flat form can be added later without breakage.

**C2. Claude gating strength: advisory vs hard.**
(a) Advisory ‚Äî keep `--dangerously-skip-permissions`, pass `--allowedTools` anyway, rely on assertions to flag violations. (b) Hard ‚Äî for gated MCP scenarios, swap to `--permission-mode dontAsk` + an explicit `--allowedTools` covering built-ins plus declared MCP tools. (c) `--disallowedTools` deny-listing ‚Äî requires knowing the server's full tool list; not statically available. ‚Üí **(a) for Phase 1** with an explicit verification task; document (b) as the upgrade.

**D. Where secrets interpolate: harness at load time vs CLI-native expansion.**
CLI-native keeps values off disk but is inconsistent (copilot/agy have nothing). ‚Üí **Hybrid**: harness interpolation as the uniform contract + fail-fast validation; adapters use native indirection where available (codex); redaction pass regardless.
