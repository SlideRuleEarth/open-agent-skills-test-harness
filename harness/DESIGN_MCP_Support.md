# Design: MCP Server Support in the Agent-Skills Test Harness

**Status: draft for discussion** ┬╖ CLI versions verified against: claude 2.1.113 ┬╖ codex 0.140.0 ┬╖ copilot 1.0.64 ┬╖ agy 1.1.1

## 1. Context and goals

We are developing a custom MCP server (SlideRule data access). The harness needs to:

1. **Let scenarios exercise MCP servers** тАФ declare a server (local stdio process or remote HTTP/SSE endpoint), have the agent under test call its tools, and assert on that usage.
2. **Control access** тАФ MCP stays hermetically OFF by default; only scenarios that explicitly declare servers get them, optionally restricted to specific tools. Credentials (API keys, bearer tokens) come from environment variables at run time and never appear in scenario YAML or archived artifacts.

All four adapters (claude, codex, copilot, antigravity) are in scope, with a phased rollout grounded in what each CLI can actually do.

### 1.1 Pre-Phase-0 state тАФ a hermeticity bug (historical; fixed below)

MCP was intentionally disabled from the start, but before Phase 0 only the Claude adapter actually enforced it (line references are to the pre-Phase-0 code):

- **claude** тАФ was already hermetic. `--strict-mcp-config` sat in `_HERMETIC` (`adapters/claude.py:63-68`) with no `--mcp-config`, so no servers loaded.
- **codex** тАФ **leaked**. The isolated HOME symlinked `~/.codex` wholesale (`isolation.py:130-149`), so any `[mcp_servers.*]` in the user's real `config.toml` loaded in every run.
- **copilot** тАФ **leaked**. `--disable-builtin-mcps` (`adapters/copilot.py:73`) only disabled the bundled GitHub server; the user's real `~/.copilot/mcp-config.json` still loaded through the symlinked HOME.
- **antigravity** тАФ **leaked** (latently). `<iso_home>/.gemini/config/mcp_config.json` was a symlink to the real file; benign at the time only because that file happened to be empty.

Fixing this was Phase 0, worth doing regardless of the rest of this design.
**Update: Phase 0 is implemented** (revised across several review rounds тАФ see ┬з8 for the mechanics). Findings from implementation that corrected this design, all verified against the installed CLIs: codex's `-c mcp_servers={}` does **not** clear the persisted table (deep-merge), disabling a name codex doesn't load *breaks* the run, its plugins are a separate MCP channel, and its effective server set depends on the run's **cwd and env** (trusted-project `.codex/config.toml`, `$CODEX_HOME`) тАФ so enumeration must run in the child's exact context; copilot/agy each have additional channels beyond the single user config file (plugins тАФ including copilot's `config.json` `cache_path` records and agy's `plugins.json` тАФ workspace configs across agy's *four* customization roots, `COPILOT_HOME`, and copilot's Windows ODR registry, which is a **fail-closed gate** тАФ copilot executes the registry-advertised command itself, so a populated gate can't be soundly pre-enumerated and instead fails the run closed), and copilot rejects `{}` as an MCP config (┬з2).

## 2. Per-CLI MCP capability survey

Everything marked **(verified)** was checked against the installed CLI's help output or embedded docs; **(inferred)** items are documented behavior not confirmable from `--help` and carry a verification task (┬з9).

### claude 2.1.113
- `--mcp-config <configs...>` тАФ loads servers from JSON files or inline strings (verified). Paired with the already-passed `--strict-mcp-config` ("only use MCP servers from --mcp-config"), declared servers become the *only* servers тАФ hermetic opt-in for free.
- Tool gating: `--allowedTools` / `--disallowedTools`; MCP tools are named `mcp__<server>__<tool>`; `mcp__<server>` allows the whole server (flags verified; naming inferred).
- Transports: stdio (`command`/`args`/`env`) and remote (`"type": "http"|"sse"`, `url`, `headers`) in the `mcpServers` JSON (inferred).
- Caveat: the harness runs with `--dangerously-skip-permissions` (`adapters/claude.py:100-101`); `--allowedTools` is a *permission* allowlist and is likely moot under it (inferred тАФ see ┬з6 decision C2).

### codex 0.140.0
- No MCP flag on `exec`; the mechanism is `-c key=value` dotted-path TOML overrides, valid on `exec` (verified).
- **`-c` deep-merges with the persisted config тАФ it cannot clear a table** (verified 0.140.0: with `[mcp_servers.review_fixture]` in config.toml, `-c mcp_servers={}` leaves the server enabled in `codex mcp list --json`). Per-server `-c mcp_servers.<name>.enabled=false` *does* disable it (verified) тАФ so hermetic-off requires enumerating configured names, and Phase 1's "declared servers only" likewise means disabling every user server not in the declared set.
- **Enumeration must match codex's effective view exactly**: a `-c тАжenabled=false` for a name codex does NOT load creates an incomplete top-level entry and the run dies with "invalid transport" (verified). The **only** enumerator is codex itself (`codex --disable plugins mcp list --json`) тАФ there is no offline config parse: a hand-parsed subset of one config.toml misses codex's system/managed layers, and a later successful main invocation could then launch an un-disabled server, so any failure of the CLI enumeration **fails closed**. Profiles don't contribute: the legacy `profile = "x"` config key is rejected outright by 0.140.0 ("no longer supported; use `--profile`"), the harness never passes `--profile`, and inactive `[profiles.*]` tables are ignored (all verified).
- **The effective view depends on the run's cwd and env** (verified 0.140.0): a *trusted* project contributes its `<git root>/.codex/config.toml` servers (untrusted projects don't; the root is resolved via git, so a subdir cwd inside a trusted repo still loads it), and `$CODEX_HOME` moves the global config. So enumeration runs `mcp list` with the **child's exact cwd and env** (exec.execute() computes the subprocess env *before* argv and hands it to the adapter via `RunOptions.effective_env`; probes get their own fresh workspace + env). Its JSON (an array of objects with a string `name`) is validated strictly тАФ `null`, `{"servers": [тАж]}` or any other drifted shape **fails closed** rather than reading as "no servers". Results are **not cached**: the effective view depends on the full execution context (resolved binary/PATH, git-root state, system/managed config), more than any home+cwd+mtime key could capture, so a stale entry could disable the wrong set.
- **Managed/MDM config can outrank the kill-switch, so the overrides are post-verified**: a higher-precedence configuration layer can override a `-c тАжenabled=false` (and `--disable plugins`). After building the disable overrides the adapter re-runs codex's own enumeration **with the overrides applied**, in the same cwd/env, and confirms every server is reported disabled (an entry counts as enabled unless it carries `"enabled": false`). Any server still enabled тАФ or any failure to obtain that post-override view тАФ **fails closed**.
- **Plugins are an MCP channel of their own**: a codex plugin can ship `.mcp.json` servers that never appear in config.toml, and disabling them by name would hit the incomplete-entry failure above. `plugins` is a stable feature flag; `--disable plugins` (valid at top level, verified) removes the channel wholesale тАФ plugin-provided skills go with it.
- Server names are restricted to `[A-Za-z0-9_-]` by `codex mcp add` (verified), i.e. always TOML-bare-key-safe in a `-c` dotted path. The `-c` parser cannot address *quoted* key segments (verified: it errors with "failed to load configuration"), so a hand-edited exotic name fails closed rather than loading.
- **`RunOptions.extra_args` are a config channel of their own**: appended after the kill-switch overrides, a `-c`/`--config` token there outranks them (verified 0.140.0: the later of two duplicate `-c mcp_servers.<n>.enabled=` overrides wins), and `--profile`/`-p`, `--cd`/`-C`, or `--enable` switches which configuration codex resolves тАФ all after the post-verify has run. `build_argv` fails closed on every spelling of those flags (standard runner cells never populate `extra_args`; only the programmatic API can).
- No feature-level kill-switch: `codex features list` has no `mcp` entry (verified).
- Config vocabulary (verified via `codex mcp add --help` and binary strings): stdio `command`/`args`/`env`; remote `url` (streamable HTTP) with `bearer_token_env_var`, `http_headers`, `env_http_headers` (header value taken from a named env var); `enabled_tools`/`disabled_tools`; `startup_timeout_sec`/`tool_timeout_sec`.
- `bearer_token_env_var` is notable: the secret never appears in argv or on disk тАФ only the env-var *name* does.
- Gap: no legacy SSE support surfaced anywhere (inferred gap).
- The harness parser already normalizes `mcp_tool_call` items into TOOL_CALL events (`adapters/codex.py:168-195`).

### copilot 1.0.64
- `--additional-mcp-config <json>` тАФ JSON string or `@file`, repeatable, augments `~/.copilot/mcp-config.json` for the session (verified).
- Per-server vocabulary via `copilot mcp add`: `--transport stdio|http|sse` (all three), `--env`, `--header`, `--timeout`, and per-server `--tools "*"|list|""` filter (verified).
- Also useful: `--secret-env-vars <names>` redacts those env values from output (verified); `COPILOT_HOME` overrides `~/.copilot` (verified); `--disable-mcp-server <name>`, `--allow-tool`/`--deny-tool` with `MyMCP(tool)` syntax (verified).
- No "ignore user mcp-config" flag exists тАФ hermeticity must come from masking the file in the isolated HOME.
- **Server sources beyond `~/.copilot/mcp-config.json`** (verified in the 1.0.64 bundle): workspace configs тАФ copilot reads `.mcp.json` / `.github/mcp.json` (the **first existing candidate per directory**, walking cwdтЖТgit root; a repo-less run reads the cwd only), `.vscode/mcp.json` having been **removed in 1.0.64**; the **built-in / feature-gated in-process servers** (`github-mcp-server`, `playwright`, `bluebird`, and the staff-gated `computer-use`, the last switchable on via `config.json` `enabledMcpServers` тАФ `--disable-builtin-mcps` documents only `github-mcp-server`); and installed plugins, whose definitions can declare `mcpServers`. The harness's workspace walk is a deliberate **conservative superset**, not CLI parity: it checks all three candidates (the removed `.vscode/mcp.json` and legacy `"servers"` spelling included) in the cwd and *every* ancestor, because over-enumerating only adds harmless disables (`--disable-mcp-server` tolerates names copilot never loads) while under-enumerating would leave a server live. The built-in names are disabled unconditionally by name, and the user `mcp-config.json` is parsed as JSONC (below). All of these must be neutralized for hermetic-off, and `COPILOT_HOME` redirects the home-based ones wholesale.
- **Plugin loading doesn't go through `installed-plugins/` alone**: plugin records live in `config.json` (`installedPlugins`, each with an absolute `cache_path` the loader follows even when `installed-plugins/` is empty тАФ verified live). `config.json` also holds auth tokens and is **full JSONC** (line, inline, *and* block comments plus trailing commas тАФ the same live-verified grammar as the user `mcp-config.json`, not just full-line `//`), so it must be *sanitized* (drop `installedPlugins`/`enabledPlugins` and `enabledMcpServers` тАФ the feature-gated built-in switch тАФ keep the rest), not replaced. An unparseable file fails closed to the neutral-but-hermetic shape.
- **The user `mcp-config.json` is JSONC too, the workspace ones are strict JSON** (live-verified via `copilot mcp list`, 1.0.64): `$COPILOT_HOME/mcp-config.json` accepts comments and trailing commas, so the enumerator parses it as JSONC and **fails closed** on an existing-but-unparseable file (copilot itself errors out on such a file); a workspace `.mcp.json` with a trailing comma is silently *ignored* by copilot, so those stay strict `json.load` (invalid тЖТ empty, matching copilot's warn-and-skip).
- **The empty mcp-config shape is `{"mcpServers": {}}`** тАФ a bare `{}` fails validation with "mcpServers: Required" (verified 1.0.64) and kills the session before execution.
- **Custom agents are an MCP channel `--disable-mcp-server` cannot be *aimed at*** (verified in the 1.0.64 bundle + installed CLI help): agent definitions come from `**/*.md` under `$COPILOT_HOME/agents`, under the `.github/agents` / `.claude/agents` convention dirs walked up from the cwd to a boundary that is the git root when git discovery finds a repo and the OS home *only when it does not* (an either/or, not the nearer of the two тАФ so a home nested **inside** a repo is walked straight past), from installed plugins, and тАФ for a git-repo cwd with a GitHub remote and auth тАФ from a *remote* org/enterprise listing fetched off the Copilot API; local frontmatter (`mcp-servers`) and remote entries alike can declare MCP servers. A selected agent's servers *are* started through `startServerOnce`, which **does** consult the session `disabledMcpServers` set тАФ but that buys the harness nothing here, because the **names are unenumerable**: they live inside agent frontmatter and, worse, inside *remote* org/enterprise listings the harness cannot read at all, so there is nothing reliable to put on `--disable-mcp-server` (parsing local frontmatter would still miss every remote entry). Disabling by name therefore cannot cover this channel, and no flag disables custom agents (`--no-custom-instructions` does not). The only off-switch that exists is the documented config setting `customAgents.defaultLocalOnly`, which short-circuits the loader before the remote listing. So: isolation masks `agents/` to an empty dir and *forces* `defaultLocalOnly` into the sanitized `config.json`; the argv layer **fails closed** on any discoverable local agent file, and on *any* run whose enumerable config doesn't provably carry the opt-out. That gate is **unconditional** тАФ not "in a git repo" тАФ because deciding it would mean *executing git*, which copilot then executes again at launch; ┬з4 explains why that two-execution gap is refused here exactly as it is for the ODR command (non-isolated runs need the opt-out set in the real `config.json`; the error says exactly that).
- **`extra_args` can reopen configuration channels after enumeration**: 1.0.64 accepts `--additional-mcp-config` (merges more servers for the session), `--agent` (selects a custom agent whose mcp-server *names* are unenumerable, see above), `--plugin-dir` (loads a plugin from any dir), the hidden `--config-dir` (repoints the whole config home), the hidden `--prefer-version` (runs a *different* cached CLI build, voiding every 1.0.64 fact this design rests on), and `-C` (changes copilot's cwd *before* discovery, invalidating the enumeration's walk). `build_argv` rejects these tokens тАФ exact, `--flag=value`, attached `-C<dir>`, **and combined short-option clusters carrying the flag** (`-sC/tmp` == `-s -C /tmp`, which a leading-`-C` rule would miss) тАФ before spending the enumeration, mirroring the codex vet.
- **Windows ODR registry is a fail-closed gate, not enumerated** (1.0.64 bundle, "ODR load/convert" pipeline): the registry value `Command` under `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Mcp` names a command line; copilot executes `<that command> mcp list`, parses the JSON `servers` array, and registers each named server. No flag/env/setting disables the mechanism and nothing lets the harness intercept copilot's load. An earlier revision executed the command *itself* and pre-disabled the resolved names, but the two executions are **independent**: a stateful or time-varying command can hand the harness listing `A` and copilot listing `B`, leaving `B`'s servers enabled тАФ matching cwd/env/bitness cannot close a two-execution gap. So a **populated gate fails the whole invocation closed** (cell, probe, judge alike); only an absent/blank gate lets a run proceed. What must still match copilot is the gate **detection**: the value is read with `KEY_QUERY_VALUE | KEY_WOW64_64KEY` тАФ the QUERY-ONLY access copilot's `RegGetValueW` uses (not the broader `KEY_READ`, which a query-only ACL would deny, making the harness reject a host copilot reads fine) plus the 64-bit view flag matching copilot's read (a 32-bit process's *default* view is the redirected WOW6432Node one, where an absent key would fake the gate "off") and judged blank-vs-populated with copilot's own **ECMAScript** `value?.trim()` + falsy test (U+FEFF is JS-space, so a lone-BOM value reads gate-off exactly as copilot sees it; U+001CтАУU+001F/U+0085 are not, so a value of only those reads gate-on). A 32-bit harness process on win32 fails closed outright regardless тАФ WOW64 file-system redirection (System32 тЖТ SysWOW64) desyncs the config FILES it reads from 64-bit copilot's view (the ODR command is copilot's to launch, not the harness's).

### antigravity (agy) 1.1.1
- No MCP flags at all (verified absence). Discovery is purely file-based: `~/.gemini/config/mcp_config.json` тАФ `{"mcpServers": {name: {command, args, env}}}` for stdio, `serverUrl` (SSE) or `url` for remote (verified via embedded docs + changelog).
- Plugins are a second file-based channel: "MCP Servers defined in `plugins/<name>/mcp_config.json`" (verified via embedded plugin docs) тАФ hermetic-off must mask that file inside every installed plugin too. `~/.gemini/config/plugins.json` can additionally register *external* plugin directories by absolute path (their MCP configs live outside every masked tree), so that registration file is masked as well.
- The **run workspace** is a third channel (verified 1.1.1 with sentinel stdio servers launched at session start): `--add-dir` exposes it, and agy recognizes **four** customization roots тАФ `.agents/`, `.agent/`, `_agents/`, `_agent/` тАФ loading, per root, `mcp_config.json`, every `plugins/<name>/mcp_config.json`, and `plugins.json` (the entries/inherits registration schema from the embedded json_configs.md тАФ an *external* plugin dir registered from a workspace `plugins.json` had its MCP server launched, verified). Ancestors of the workspace are NOT scanned (verified). A seeded scenario file could re-enable MCP, so the runner neutralizes all sixteen patterns after seeding (`workspace_config_masks` тАФ per root: `mcp_config.json`, `plugins/*/mcp_config.json` plus its dot-inclusive companion `plugins/.*/mcp_config.json` (glob's `*` skips hidden names, and agy loads dot-prefixed plugin dirs too, verified), and `plugins.json`), and model probes anchor `--add-dir` to a fresh private 0700 temp dir instead of the shared system temp root (whose planted `.agents/` would load тАФ `/tmp` is world-writable on Linux).
- **No tool-allowlist mechanism** (verified absence) тАФ documented gap.
- Transcript tool names look like `mcp_<server>_<tool>` with single underscores (inferred from binary strings).

## 3. Scenario schema extension

New optional top-level field on evals and scenarios. Absent тЗТ MCP hermetically off (the default today, post-Phase-0 for all adapters).

```yaml
mcp_servers:                              # dict[str, server]; key = server name = tool namespace
  sliderule:                              # тЖТ tools surface as mcp__sliderule__<tool>
    command: python3                      # stdio transport (XOR with url); bare names stay PATH-resolved
    args: [fixtures/sliderule_mcp.py]     # entries naming files under the scenario dir are absolutized
    env:
      SLIDERULE_DOMAIN: slideruleearth.io
      SLIDERULE_API_KEY: ${SLIDERULE_MCP_TOKEN}    # interpolated from harness process env
    tools: [atl06p, atl03x]               # optional allowlist; omit тЗТ all tools

  sliderule-remote:
    url: https://sliderule.example.com/mcp     # remote transport (XOR with command)
    transport: sse                        # http (default) | sse
    headers:
      Authorization: Bearer ${SLIDERULE_MCP_TOKEN}
```

Validation rules (added to `validate_spec`, `spec.py:237-395`, so misconfiguration fails before any tokens are spent):
- Exactly one of `command` / `url` per server; unknown keys тЖТ error.
- `${VAR}` interpolation in `env` values, `headers` values, and `url`, resolved from the harness process environment at load time; unresolvable тЖТ error naming the variable.
- Every interpolated value is registered in a per-run **secrets set** (used for artifact redaction, ┬з5.3).
- Path absolutization is selective, resolving against the spec's `base_dir()` (same convention as `files:`, `spec.py:159-182`) тАФ the agent runs in a tempdir workspace, so scenario-relative paths would otherwise break. `command` is absolutized only when it contains a path separator *and* names an existing file under `base_dir()`; a bare executable name (`python3`) is left for PATH lookup. Each `args` entry is absolutized only when it names an existing file or directory under `base_dir()`; flags and other non-path arguments pass through verbatim.
- `tools:` on an antigravity target тЖТ warning ("advisory only"); `mcp_servers` + `isolated: false` on an antigravity target тЖТ error (┬з4, agy).

### Plumbing (mirrors the existing `output_schema` / `env` pattern)

1. `EvalSpec.mcp_servers: dict` тАФ parsed in `_spec_from_raw` (`spec.py:433-500`).
2. `RunOptions.mcp_servers: Optional[dict]` and `RunOptions.mcp_scratch_dir: Optional[str]` (`adapters/base.py:27-45`).
3. Runner populates both per cell (`runner.py:345-352`). The scratch dir is a per-cell `ase-mcp-` tempdir created alongside the isolated home and deleted in the same `finally` (`runner.py:359-361`). It is deliberately **outside the workspace**: the workspace is relocated into `artifacts/**/workspace` and inlined into `report.md` (`runner.py:413-415`, `render_report`), which would archive secret-bearing config files.
4. Each adapter's `build_argv` / `env` consumes them (┬з4).

Details:

- **claude** (`adapters/claude.py:90-113`): when `opts.mcp_servers` is set, write `{"mcpServers": тАж}` to `<scratch>/mcp.json` and append `["--mcp-config", path]`. A *file*, not inline JSON тАФ argv is recorded in `result.json` (`exec.py:49`, `runner.py:495`), so inline JSON with resolved secrets would land in artifacts. Per-server `tools:` compiles to `--allowedTools` entries.
- **codex** (`adapters/codex.py`): the Phase-0 kill-switch is `--disable plugins` (the plugin MCP channel, ┬з2) plus `-c mcp_servers.<name>.enabled=false` per configured server, enumerated primarily by `codex --disable plugins mcp list --json` **run with the child's exact cwd and env** (exec.execute() hands adapters the computed subprocess env via `RunOptions.effective_env` before building argv; probe_model passes the probe's own fresh workspace/env) тАФ codex's own effective-config view, which is what keeps the disables exactly matched to loaded servers: a disable for an unloaded name kills the run, and the loaded set shifts with trusted-project `.codex/config.toml`s and `$CODEX_HOME` (┬з2). The `mcp list` JSON is strictly validated (schema drift **fails closed**, never "no servers"), results are **not cached** (the effective view depends on more than any home+cwd+mtime key can capture), and there is **no offline fallback**: any enumeration failure fails closed, because a hand-parsed subset of one config.toml misses codex's system/managed layers (┬з2). A managed/MDM layer can also outrank the `-c` overrides, so once built they are **post-verified** тАФ codex's own enumeration is re-run with the overrides applied and the run fails closed if any server is still enabled (┬з2). Riding on argv covers cells, probes, and judge runs identically (`-c mcp_servers={}` deep-merges and does not work, ┬з2), and `build_argv` rejects config-channel `extra_args` outright тАФ appended after the overrides, they would outrank or sidestep the post-verified state (┬з2). With declared servers (Phase 1): `-c mcp_servers.sliderule.command="python3"`, `-c 'mcp_servers.sliderule.args=[тАж]'`, `-c 'mcp_servers.sliderule.env={тАж}'`, `-c 'mcp_servers.sliderule.enabled_tools=[тАж]'`; remote: `url` + `bearer_token_env_var` / `env_http_headers` so HTTP secrets never materialize; user servers outside the declared set still get the per-name disables. Stdio `env` literals do pass through argv тЖТ covered by the redaction pass (┬з5.3). Writing a per-run `$CODEX_HOME/config.toml` was considered and rejected for Phase 1 (┬з6-B).

## 5. Access control

### 5.1 Opt-in (server level)
Default stays hermetic via the kill-switch column in ┬з4. ### 5.2 Tool gating (tool level)
Per-server `tools:` compiles to each CLI's native filter тАФ codex `enabled_tools` and copilot per-server `tools` are hard filters (verified). Claude: `--allowedTools` is likely advisory under `--dangerously-skip-permissions`; Phase 1 accepts advisory gating (assertions still catch off-limits usage), with a documented upgrade path (┬з6-C2) if hard enforcement matters. agy: no mechanism; validation warns.

### 5.3 Credentials
- `${VAR}` interpolation from the harness process env only; fail-fast at validation; values never committed in YAML.
- Prefer CLI-native env-var **indirection** where it exists (codex `bearer_token_env_var`, `env_http_headers`) тАФ the value then never touches disk or argv at all.
- Scratch config files created `0600`, outside the workspace, deleted post-run.
- **Redaction pass**: every interpolated secret value is scrubbed from `stdout.jsonl`, `stderr.txt`, `events.json`, `result.json` (including recorded `argv`), and `report.md` in `_write_artifacts` / `render_report` (`runner.py:488-495`, `runner.py:766`). This also covers the case no CLI flag can: a tool *result* echoing a token back into the transcript.

## 6. Decision points

**A. Config delivery for claude/copilot: inline argv JSON vs scratch file vs workspace `.mcp.json`.**
Inline JSON leaks resolved secrets into `result.json` (argv is archived) and needs shell-safe quoting. Workspace files are archived and inlined into `report.md`; claude ignores workspace `.mcp.json` under `--strict-mcp-config` anyway. тЖТ **Scratch file** in the per-cell tempdir, deleted post-run.

**B. Codex delivery: `-c` overrides vs materialized `$CODEX_HOME/config.toml`.**
`-c` needs no isolation surgery, keeps HTTP secrets env-indirect natively, and its one exposure (stdio env literals in argv) is covered by redaction. A materialized config home means converting `.codex` from a wholesale symlink into a masked dir with a merged TOML (preserving auth) тАФ heavier and riskier. тЖТ **`-c` overrides**, with config-home materialization as the fallback if TOML array/inline-table `-c` values prove unreliable (verification task).

**C1. Allowlist shape: flat top-level `allowed_tools: [mcp__sliderule__*]` vs per-server `tools:`.**
Flat matches claude's flag but forces claude's naming onto codex/copilot (whose native filters take bare tool names per server) and repeats server names. тЖТ **Per-server `tools:`**; adapters compile to their native form. A flat form can be added later without breakage.

**C2. Claude gating strength: advisory vs hard.**
(a) Advisory тАФ keep `--dangerously-skip-permissions`, pass `--allowedTools` anyway, rely on assertions to flag violations. (b) Hard тАФ for gated MCP scenarios, swap to `--permission-mode dontAsk` + an explicit `--allowedTools` covering built-ins plus declared MCP tools. (c) `--disallowedTools` deny-listing тАФ requires knowing the server's full tool list; not statically available. тЖТ **(a) for Phase 1** with an explicit verification task; document (b) as the upgrade.

**D. Where secrets interpolate: harness at load time vs CLI-native expansion.**
CLI-native keeps values off disk but is inconsistent (copilot/agy have nothing). тЖТ **Hybrid**: harness interpolation as the uniform contract + fail-fast validation; adapters use native indirection where available (codex); redaction pass regardless.
