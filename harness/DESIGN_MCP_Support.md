# Design: MCP Server Support in the Agent-Skills Test Harness

**Status: draft for discussion** · CLI versions verified against: claude 2.1.113 · codex 0.140.0 · copilot 1.0.64 · agy 1.1.1

## 1. Context and goals

We are developing a custom MCP server (SlideRule data access). The harness needs to:

1. **Let scenarios exercise MCP servers** — declare a server (local stdio process or remote HTTP/SSE endpoint), have the agent under test call its tools, and assert on that usage.
2. **Control access** — MCP stays hermetically OFF by default; only scenarios that explicitly declare servers get them, optionally restricted to specific tools. Credentials (API keys, bearer tokens) come from environment variables at run time and never appear in scenario YAML or archived artifacts. "Off by default" is enforced by *prevention* everywhere except a residual window on copilot **and codex**, where the CLI re-reads its own discovery state after launch and the harness cannot freeze that state; there the guarantee is narrowed to **no run reported clean had its hermeticity premise broken** — and on **codex that narrowed guarantee is itself qualified**: a server that started and was removed before the post-run check, without the model calling it, is undetectable (see the update below). §4 states what the copilot case does and does not cover.

> **Update — codex's launch window is now NARROWED, and one case stays open.** The Phase 0 text below describes codex's `_verify_all_mcp_disabled` as a "post-verify". That name is misleading: it runs from `build_argv`, so it is post-*enumeration* and pre-*launch*. A server added to config between argv construction and codex reading its config at startup was never named on the disable set, loaded live, and nothing checked afterwards. `CodexAdapter.verify_post_run` now re-enumerates after the run and fails closed on any server configured but not disabled on the launched argv, and additionally reads the run's own stream for `mcp_tool_call` items.
>
> **Residual undetectable gap (codex only).** A server added after argv was built, started by codex, and removed again before the post-run enumeration is invisible to *both* checks whenever the model never calls one of its tools — the stream is silent because codex emits nothing when a server starts, and re-enumeration sees the restored configuration. Because it is precisely the *idle* server that escapes, and a server need not be called to act (the sentinel received `initialize`/`tools/list` and ran as the agent for the whole session), this is a real hole rather than a technicality. **For codex, a clean verification means "no leak was detected", not "no server ran".** An earlier draft of this section claimed the two checks covered each other's blind spots; that holds for copilot, whose witness names every server it brought up, and is false for codex — review caught the over-claim.
>
> Closing it requires *prevention*: a materialized private configuration for the child, so no file an outside writer can reach is consulted mid-run. That is a real change to codex's isolation story — `$CODEX_HOME` currently mirrors the user's, which is what keeps auth working — and is not attempted here. Tracked in `TODO_Version_Provenance.md`.
>
> **codex's stream evidence is much weaker than copilot's, and the asymmetry is load-bearing.** Verified live against a sentinel stdio MCP server: codex emits **no event when it starts a server** — the sentinel was launched and exchanged `initialize`/`tools/list` while the stream showed its usual four events. The only MCP trace is an `mcp_tool_call` item, emitted only if the model actually calls a tool. So for codex, presence proves a leak but absence proves nothing, and the re-enumeration half carries the argument — the reverse of copilot, whose `session.mcp_servers_loaded` positively witnesses a clean run.
>
> **`--ask-for-approval never` is not an MCP control.** The same test showed it cancelling the sentinel's tool *call* ("user cancelled MCP tool call") while the server process ran as the agent for the entire session. Only the `-c mcp_servers.<name>.enabled=false` overrides stop the process starting (verified: with them applied, the sentinel never ran at all).

All four adapters (claude, codex, copilot, antigravity) are in scope, with a phased rollout grounded in what each CLI can actually do.

### 1.1 Pre-Phase-0 state — a hermeticity bug (historical; fixed below)

MCP was intentionally disabled from the start, but before Phase 0 only the Claude adapter actually enforced it (line references are to the pre-Phase-0 code):

- **claude** — was already hermetic. `--strict-mcp-config` sat in `_HERMETIC` (`adapters/claude.py:63-68`) with no `--mcp-config`, so no servers loaded.
- **codex** — **leaked**. The isolated HOME symlinked `~/.codex` wholesale (`isolation.py:130-149`), so any `[mcp_servers.*]` in the user's real `config.toml` loaded in every run.
- **copilot** — **leaked**. `--disable-builtin-mcps` (`adapters/copilot.py:73`) only disabled the bundled GitHub server; the user's real `~/.copilot/mcp-config.json` still loaded through the symlinked HOME.
- **antigravity** — **leaked** (latently). `<iso_home>/.gemini/config/mcp_config.json` was a symlink to the real file; benign at the time only because that file happened to be empty.

Fixing this was Phase 0, worth doing regardless of the rest of this design.

**Update: Phase 0 does not support Windows, and refuses to run there.** The design below
works out copilot's Windows ODR gate, WOW64 views and `%USERPROFILE%` resolution in
detail, and all of that stands — but it is downstream of a guarantee the harness cannot
currently make on that platform at all. Every hermeticity claim here rests on being able
to kill the whole process tree the agent creates, and on Windows the only mechanism for
that (a Job Object) can be assigned only *after* `CreateProcess` returns, while the OS
associates a member's **future** children only. A grandchild spawned in that window —
an MCP server started as the first thing the agent does — is permanently outside the job.
Job-at-creation via `PROC_THREAD_ATTRIBUTE_JOB_LIST` is unreachable through stdlib
`Popen` (CPython honours `lpAttributeList` for `handle_list` and nothing else), so
closing it needs a bespoke `CreateProcess` or a launcher shim, plus a Windows host to
verify on — neither of which exists yet. Rather than ship a guarantee that is silently
unsound on one platform, `exec.run_captured` refuses win32 **before it spawns anything**
(`UnsupportedPlatform`, an `OSError` subclass so existing callers fail closed on it
unchanged). Refusing after the `Popen` would refuse nothing — the grandchild being
guarded against starts in the child's first instants, so a check that runs once the child
exists has already let it happen. The Job Object
code stays in place, correct and tested against a stubbed kernel32, because it is still
the right sweep for the child that did start and because the follow-up that adds
job-at-creation should change one predicate rather than start from nothing.

**Update: Phase 0 is implemented** (revised across several review rounds — see §8 for the mechanics). Findings from implementation that corrected this design, all verified against the installed CLIs: codex's `-c mcp_servers={}` does **not** clear the persisted table (deep-merge), disabling a name codex doesn't load *breaks* the run, its plugins are a separate MCP channel, and its effective server set depends on the run's **cwd and env** (trusted-project `.codex/config.toml`, `$CODEX_HOME`) — so enumeration must run in the child's exact context; copilot/agy each have additional channels beyond the single user config file (plugins — including copilot's `config.json` `cache_path` records and agy's `plugins.json` — workspace configs across agy's *four* customization roots, `COPILOT_HOME`, and copilot's Windows ODR registry, which is a **fail-closed gate** — copilot executes the registry-advertised command itself, so a populated gate can't be soundly pre-enumerated and instead fails the run closed), and copilot rejects `{}` as an MCP config (§2).

## 2. Per-CLI MCP capability survey

Everything marked **(verified)** was checked against the installed CLI's help output or embedded docs; **(inferred)** items are documented behavior not confirmable from `--help` and carry a verification task (§9).

### claude 2.1.113
- `--mcp-config <configs...>` — loads servers from JSON files or inline strings (verified). Paired with the already-passed `--strict-mcp-config` ("only use MCP servers from --mcp-config"), declared servers become the *only* servers — hermetic opt-in for free.
- Tool gating: MCP tools are named `mcp__<server>__<tool>` (**verified** live against the echo fixture, 2.1.113 — §6-C2). `--disallowedTools` is the mechanism that bites: it removes tools from the advertised surface, accepts wildcards (`mcp__<server>__*`), and **beats** `--allowedTools` when both name the same tool (all verified). `--allowedTools` gates nothing under the flags this harness passes (§6-C2). The bare `mcp__<server>` whole-server prefix form was **not** exercised — the harness emits explicit names and `*` wildcards only, so it stays unverified rather than assumed.
- Transports in the `mcpServers` JSON: stdio `{"command":…, "args":[…]}` — **verified** end-to-end against the echo fixture, server reported `connected` in the init event and its tool executed; remote (`"type": "http"|"sse"`, `url`, `headers`) still **inferred** (§9.1).
- The harness runs with `--dangerously-skip-permissions` (`adapters/claude.py:100-101`), under which `--allowedTools` is not "likely moot" but **entirely inert** for MCP tools: the unlisted tool stays advertised, the model calls it, it executes (**measured** 2026-07-22, not inferred — §6-C2). The one flag that makes an allowlist bite, `--permission-mode dontAsk`, makes it bite on *every* tool including built-ins — rejected on comparability grounds, not on strength (§6-C2).

### codex 0.140.0
- No MCP flag on `exec`; the mechanism is `-c key=value` dotted-path TOML overrides, valid on `exec` (verified).
- **`-c` deep-merges with the persisted config — it cannot clear a table** (verified 0.140.0: with `[mcp_servers.review_fixture]` in config.toml, `-c mcp_servers={}` leaves the server enabled in `codex mcp list --json`). Per-server `-c mcp_servers.<name>.enabled=false` *does* disable it (verified) — so hermetic-off requires enumerating configured names, and Phase 1's "declared servers only" likewise means disabling every user server not in the declared set.
- **Enumeration must match codex's effective view exactly**: a `-c …enabled=false` for a name codex does NOT load creates an incomplete top-level entry and the run dies with "invalid transport" (verified). The **only** enumerator is codex itself (`codex --disable plugins mcp list --json`) — there is no offline config parse: a hand-parsed subset of one config.toml misses codex's system/managed layers, and a later successful main invocation could then launch an un-disabled server, so any failure of the CLI enumeration **fails closed**. Profiles don't contribute: the legacy `profile = "x"` config key is rejected outright by 0.140.0 ("no longer supported; use `--profile`"), the harness never passes `--profile`, and inactive `[profiles.*]` tables are ignored (all verified).
- **The effective view depends on the run's cwd and env** (verified 0.140.0): a *trusted* project contributes its `<git root>/.codex/config.toml` servers (untrusted projects don't; the root is resolved via git, so a subdir cwd inside a trusted repo still loads it), and `$CODEX_HOME` moves the global config. So enumeration runs `mcp list` with the **child's exact cwd and env** (exec.execute() computes the subprocess env *before* argv and hands it to the adapter via `RunOptions.effective_env`; probes get their own fresh workspace + env). Its JSON (an array of objects with a string `name`) is validated strictly — `null`, `{"servers": […]}` or any other drifted shape **fails closed** rather than reading as "no servers". Results are **not cached**: the effective view depends on the full execution context (resolved binary/PATH, git-root state, system/managed config), more than any home+cwd+mtime key could capture, so a stale entry could disable the wrong set.
- **Managed/MDM config can outrank the kill-switch, so the overrides are post-verified**: a higher-precedence configuration layer can override a `-c …enabled=false` (and `--disable plugins`). After building the disable overrides the adapter re-runs codex's own enumeration **with the overrides applied**, in the same cwd/env, and confirms every server is reported disabled (an entry counts as enabled unless it carries `"enabled": false`). Any server still enabled — or any failure to obtain that post-override view — **fails closed**.
- **Plugins are an MCP channel of their own**: a codex plugin can ship `.mcp.json` servers that never appear in config.toml, and disabling them by name would hit the incomplete-entry failure above. `plugins` is a stable feature flag; `--disable plugins` (valid at top level, verified) removes the channel wholesale — plugin-provided skills go with it.
- Server names are restricted to `[A-Za-z0-9_-]` by `codex mcp add` (verified), i.e. always TOML-bare-key-safe in a `-c` dotted path. The `-c` parser cannot address *quoted* key segments (verified: it errors with "failed to load configuration"), so a hand-edited exotic name fails closed rather than loading.
- **`RunOptions.extra_args` are a config channel of their own**: appended after the kill-switch overrides, a `-c`/`--config` token there outranks them (verified 0.140.0: the later of two duplicate `-c mcp_servers.<n>.enabled=` overrides wins), and `--profile`/`-p`, `--cd`/`-C`, or `--enable` switches which configuration codex resolves — all after the post-verify has run. `build_argv` fails closed on every spelling of those flags (standard runner cells never populate `extra_args`; only the programmatic API can).
- No feature-level kill-switch: `codex features list` has no `mcp` entry (verified).
- Config vocabulary (verified via `codex mcp add --help` and binary strings): stdio `command`/`args`/`env`; remote `url` (streamable HTTP) with `bearer_token_env_var`, `http_headers`, `env_http_headers` (header value taken from a named env var); `enabled_tools`/`disabled_tools`; `startup_timeout_sec`/`tool_timeout_sec`.
- `bearer_token_env_var` is notable: the secret never appears in argv or on disk — only the env-var *name* does.
- Gap: no legacy SSE support surfaced anywhere (inferred gap).
- The harness parser already normalizes `mcp_tool_call` items into TOOL_CALL events (`adapters/codex.py:168-195`).

### copilot 1.0.64
- `--additional-mcp-config <json>` — JSON string or `@file`, repeatable, augments `~/.copilot/mcp-config.json` for the session (verified).
- Per-server vocabulary via `copilot mcp add`: `--transport stdio|http|sse` (all three), `--env`, `--header`, `--timeout`, and per-server `--tools "*"|list|""` filter (verified).
- Also useful: `--secret-env-vars <names>` redacts those env values from output (verified); `COPILOT_HOME` overrides `~/.copilot` (verified); `--disable-mcp-server <name>`, `--allow-tool`/`--deny-tool` with `MyMCP(tool)` syntax (verified).
- No "ignore user mcp-config" flag exists — hermeticity must come from masking the file in the isolated HOME.
- **Server sources beyond `~/.copilot/mcp-config.json`** (verified in the 1.0.64 bundle): workspace configs — copilot reads `.mcp.json` / `.github/mcp.json` (the **first existing candidate per directory**, walking cwd→git root; a repo-less run reads the cwd only), `.vscode/mcp.json` having been **removed in 1.0.64**; the **built-in / feature-gated in-process servers** (`github-mcp-server`, `playwright`, `bluebird`, and the staff-gated `computer-use`, the last switchable on via `config.json` `enabledMcpServers` — `--disable-builtin-mcps` documents only `github-mcp-server`); and installed plugins, whose definitions can declare `mcpServers`. The harness's workspace walk is a deliberate **conservative superset**, not CLI parity: it checks all three candidates (the removed `.vscode/mcp.json` and legacy `"servers"` spelling included) in the cwd and *every* ancestor, because over-enumerating only adds harmless disables (`--disable-mcp-server` tolerates names copilot never loads) while under-enumerating would leave a server live. The built-in names are disabled unconditionally by name, and the user `mcp-config.json` is parsed as JSONC (below). All of these must be neutralized for hermetic-off, and `COPILOT_HOME` redirects the home-based ones wholesale.
- **Plugin loading doesn't go through `installed-plugins/` alone**: plugin records live in `config.json` (`installedPlugins`, each with an absolute `cache_path` the loader follows even when `installed-plugins/` is empty — verified live). `config.json` also holds auth tokens and is **full JSONC** (line, inline, *and* block comments plus trailing commas — the same live-verified grammar as the user `mcp-config.json`, not just full-line `//`), so it must be *sanitized* (drop `installedPlugins`/`enabledPlugins` and `enabledMcpServers` — the feature-gated built-in switch — keep the rest), not replaced. An unparseable file fails closed to the neutral-but-hermetic shape.
- **The user `mcp-config.json` is JSONC too, the workspace ones are strict JSON** (live-verified via `copilot mcp list`, 1.0.64): `$COPILOT_HOME/mcp-config.json` accepts comments and trailing commas, so the enumerator parses it as JSONC and **fails closed** on an existing-but-unparseable file (copilot itself errors out on such a file); a workspace `.mcp.json` with a trailing comma is silently *ignored* by copilot, so those stay strict `json.load` (invalid → empty, matching copilot's warn-and-skip).
- **The empty mcp-config shape is `{"mcpServers": {}}`** — a bare `{}` fails validation with "mcpServers: Required" (verified 1.0.64) and kills the session before execution.
- **Custom agents are an MCP channel `--disable-mcp-server` cannot be *aimed at*** (verified in the 1.0.64 bundle + installed CLI help): agent definitions come from `**/*.md` under `$COPILOT_HOME/agents`, under the `.github/agents` / `.claude/agents` convention dirs walked up from the cwd to a boundary that is the git root when git discovery finds a repo and the OS home *only when it does not* (an either/or, not the nearer of the two — so a home nested **inside** a repo is walked straight past), from installed plugins, and — for a git-repo cwd with a GitHub remote and auth — from a *remote* org/enterprise listing fetched off the Copilot API; local frontmatter (`mcp-servers`) and remote entries alike can declare MCP servers. A selected agent's servers *are* started through `startServerOnce`, which **does** consult the session `disabledMcpServers` set — but that buys the harness nothing here, because the **names are unenumerable**: they live inside agent frontmatter and, worse, inside *remote* org/enterprise listings the harness cannot read at all, so there is nothing reliable to put on `--disable-mcp-server` (parsing local frontmatter would still miss every remote entry). Disabling by name therefore cannot cover this channel, and no flag disables custom agents (`--no-custom-instructions` does not). The only off-switch that exists is the documented config setting `customAgents.defaultLocalOnly`, which short-circuits the loader before the remote listing. That setting lives in **two** files, not one: 1.0.64 migrates user settings between `config.json` and `settings.json` at startup (verified live — an opt-out written into `config.json` was applied for that run and then moved into a `settings.json` copilot created, value intact). So the harness reads the opt-out from either file (a *disagreement* between them fails closed: which wins is decided inside a native module, not readable JS) and masks **both** under isolation with the same sanitizer — otherwise a key stripped from one rides back in through the other, which the overlay symlinks wholesale unless it is declared a mask. Reading only `config.json`, as an earlier revision did, also made the opt-out *evaporate*: copilot relocates the key, so the next run finds it missing and fails closed telling a non-isolated user to set the key they already set. One ordering detail matters for the harness: copilot **executes** `git rev-parse` (child PATH) to pick that boundary *before* it globs the agent dirs, so anything that runs as `git` there acts inside the harness's preflight-to-launch window (§4). So: isolation masks `agents/` to an empty dir and *forces* `defaultLocalOnly` into the sanitized `config.json`; the argv layer **fails closed** on any discoverable local agent file, and on *any* run whose enumerable config doesn't provably carry the opt-out. That gate is **unconditional** — not "in a git repo" — because deciding it would mean *executing git*, which copilot then executes again at launch; §4 explains why that two-execution gap is refused here exactly as it is for the ODR command (non-isolated runs need the opt-out set in the real `config.json`; the error says exactly that).
- **`extra_args` can reopen configuration channels after enumeration**: 1.0.64 accepts `--additional-mcp-config` (merges more servers for the session), `--agent` (selects a custom agent whose mcp-server *names* are unenumerable, see above), `--plugin-dir` (loads a plugin from any dir), the hidden `--config-dir` (repoints the whole config home), the hidden `--prefer-version` (runs a *different* cached CLI build, voiding every 1.0.64 fact this design rests on), and `-C` (changes copilot's cwd *before* discovery, invalidating the enumeration's walk). `build_argv` rejects these tokens — exact, `--flag=value`, attached `-C<dir>`, **and combined short-option clusters carrying the flag** (`-sC/tmp` == `-s -C /tmp`, which a leading-`-C` rule would miss) — before spending the enumeration, mirroring the codex vet.
- **Windows ODR registry is a fail-closed gate, not enumerated** (1.0.64 bundle, "ODR load/convert" pipeline): the registry value `Command` under `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Mcp` names a command line; copilot executes `<that command> mcp list`, parses the JSON `servers` array, and registers each named server. No flag/env/setting disables the mechanism and nothing lets the harness intercept copilot's load. An earlier revision executed the command *itself* and pre-disabled the resolved names, but the two executions are **independent**: a stateful or time-varying command can hand the harness listing `A` and copilot listing `B`, leaving `B`'s servers enabled — matching cwd/env/bitness cannot close a two-execution gap. So a **populated gate fails the whole invocation closed** (cell, probe, judge alike); only an absent/blank gate lets a run proceed. What must still match copilot is the gate **detection**: the value is read with `KEY_QUERY_VALUE | KEY_WOW64_64KEY` — the QUERY-ONLY access copilot's `RegGetValueW` uses (not the broader `KEY_READ`, which a query-only ACL would deny, making the harness reject a host copilot reads fine) plus the 64-bit view flag matching copilot's read (a 32-bit process's *default* view is the redirected WOW6432Node one, where an absent key would fake the gate "off") and judged blank-vs-populated with copilot's own **ECMAScript** `value?.trim()` + falsy test (U+FEFF is JS-space, so a lone-BOM value reads gate-off exactly as copilot sees it; U+001C–U+001F/U+0085 are not, so a value of only those reads gate-on). A 32-bit harness process on win32 fails closed outright regardless — WOW64 file-system redirection (System32 → SysWOW64) desyncs the config FILES it reads from 64-bit copilot's view (the ODR command is copilot's to launch, not the harness's).

### antigravity (agy) 1.1.1
- No MCP flags at all (verified absence). Discovery is purely file-based: `~/.gemini/config/mcp_config.json` — `{"mcpServers": {name: {command, args, env}}}` for stdio, `serverUrl` (SSE) or `url` for remote (verified via embedded docs + changelog).
- Plugins are a second file-based channel: "MCP Servers defined in `plugins/<name>/mcp_config.json`" (verified via embedded plugin docs) — hermetic-off must mask that file inside every installed plugin too. `~/.gemini/config/plugins.json` can additionally register *external* plugin directories by absolute path (their MCP configs live outside every masked tree), so that registration file is masked as well.
- The **run workspace** is a third channel (verified 1.1.1 with sentinel stdio servers launched at session start): `--add-dir` exposes it, and agy recognizes **four** customization roots — `.agents/`, `.agent/`, `_agents/`, `_agent/` — loading, per root, `mcp_config.json`, every `plugins/<name>/mcp_config.json`, and `plugins.json` (the entries/inherits registration schema from the embedded json_configs.md — an *external* plugin dir registered from a workspace `plugins.json` had its MCP server launched, verified). Ancestors of the workspace are NOT scanned (verified). A seeded scenario file could re-enable MCP, so the runner neutralizes all sixteen patterns after seeding (`workspace_config_masks` — per root: `mcp_config.json`, `plugins/*/mcp_config.json` plus its dot-inclusive companion `plugins/.*/mcp_config.json` (glob's `*` skips hidden names, and agy loads dot-prefixed plugin dirs too, verified), and `plugins.json`), and model probes anchor `--add-dir` to a fresh private 0700 temp dir instead of the shared system temp root (whose planted `.agents/` would load — `/tmp` is world-writable on Linux).
- **No tool-allowlist mechanism** (verified absence) — documented gap.
- Transcript tool names look like `mcp_<server>_<tool>` with single underscores (inferred from binary strings).

## 3. Scenario schema extension

New optional top-level field on evals and scenarios. Absent ⇒ MCP hermetically off (the default today, post-Phase-0 for all adapters).

```yaml
mcp_servers:                              # dict[str, server]; key = server name = tool namespace
  sliderule:                              # → tools surface as mcp__sliderule__<tool>
    command: python3                      # stdio transport (XOR with url); bare names stay PATH-resolved
    args: [fixtures/sliderule_mcp.py]     # entries naming files under the scenario dir are absolutized
    env:
      SLIDERULE_DOMAIN: slideruleearth.io
      SLIDERULE_API_KEY: ${SLIDERULE_MCP_TOKEN}    # interpolated from harness process env
    tools: [atl06p, atl03x]               # optional allowlist; omit ⇒ all tools

  sliderule-remote:
    url: https://sliderule.example.com/mcp     # remote transport (XOR with command)
    transport: sse                        # http (default) | sse
    headers:
      Authorization: Bearer ${SLIDERULE_MCP_TOKEN}
```

Validation rules (added to `validate_spec`, `spec.py:237-395`, so misconfiguration fails before any tokens are spent):
- Exactly one of `command` / `url` per server; unknown keys → error.
- `${VAR}` interpolation in `env` values, `headers` values, and `url`, resolved from the harness process environment at load time; unresolvable → error naming the variable.
- Every interpolated value is registered in a per-run **secrets set** (used for artifact redaction, §5.3).
- Path absolutization is selective, resolving against the spec's `base_dir()` (same convention as `files:`, `spec.py:159-182`) — the agent runs in a tempdir workspace, so scenario-relative paths would otherwise break. `command` is absolutized only when it contains a path separator *and* names an existing file under `base_dir()`; a bare executable name (`python3`) is left for PATH lookup. Each `args` entry is absolutized only when it names an existing file or directory under `base_dir()`; flags and other non-path arguments pass through verbatim.
- The **CLI-native filter spellings** — `allowedTools`, `enabledTools`, `enabled_tools`, `disabled_tools` — are **rejected** on a server. `tools:` (above) is the harness's portable field; the native spellings are what claude accepts and then **silently ignores** (measured, §6-C2), so passing one through would write a key that does nothing quietly. The error names `tools:` as the replacement.
- `tools:` on an antigravity target → warning ("advisory only"); `mcp_servers` + `isolated: false` on an antigravity target → error (§4, agy).
- **`tools:` must never be accepted where it is not enforced.** Each adapter either compiles it to a real filter or the spec fails validation naming the adapter — an accepted-and-ignored allowlist is the exact failure mode claude's own config has, and reproducing it in the harness's schema would be worse, since the scenario author has no way to notice. Where enforcement is snapshot-prevention plus detection rather than hard prevention (claude, §6-C2), that is a documented property of the field, not a reason to reject it.

### Plumbing (mirrors the existing `output_schema` / `env` pattern)

1. `EvalSpec.mcp_servers: dict` — parsed in `_spec_from_raw` (`spec.py:433-500`).
2. `RunOptions.mcp_servers: Optional[dict]` and `RunOptions.mcp_scratch_dir: Optional[str]` (`adapters/base.py:27-45`).
3. Runner populates both per cell (`runner.py:345-352`). The scratch dir is a per-cell `ase-mcp-` tempdir created alongside the isolated home and deleted in the same `finally` (`runner.py:359-361`). It is deliberately **outside the workspace**: the workspace is relocated into `artifacts/**/workspace` and inlined into `report.md` (`runner.py:413-415`, `render_report`), which would archive secret-bearing config files.

## 5. Access control

### 5.1 Opt-in (server level)
Default stays hermetic via the kill-switch column in §4. When `mcp_servers` is declared, only those servers are reachable: claude's `--strict-mcp-config`; codex's per-name `enabled=false` disables for every user server outside the declared set (a full-table `-c mcp_servers={}` override was disproven — it deep-merges, §2) plus `--disable plugins`; copilot/agy's masked user config.

### 5.2 Tool gating (tool level)
Per-server `tools:` compiles to each CLI's native filter — codex `enabled_tools` and copilot per-server `tools` are hard filters (verified). Claude: `--allowedTools` does **nothing at all** to MCP tools under `--dangerously-skip-permissions` — not a soft filter, not advice; the tool stays advertised and executes (measured, not inferred — §6-C2). The mechanism is `--disallowedTools` on the *complement*, which removes tools from the advertised surface entirely, paired with a fail-closed post-run subset check against the run's own init event. That pairing is **prevention for the enumerated snapshot plus fail-closed detection of drift** — not hard prevention against a server that answers the enumerator and claude differently (§6-C2; C3 is the mechanism that would be). agy: no mechanism; validation warns.

### 5.3 Credentials
- `${VAR}` interpolation from the harness process env only; fail-fast at validation; values never committed in YAML.
- Prefer CLI-native env-var **indirection** where it exists (codex `bearer_token_env_var`, `env_http_headers`) — the value then never touches disk or argv at all.
- Scratch config files created `0600`, outside the workspace, deleted post-run.
- **Redaction pass**: every interpolated secret value is scrubbed from `stdout.jsonl`, `stderr.txt`, `events.json`, `result.json` (including recorded `argv`), `report.md`, the archived `workspace/`, and the run-level `summary.json` / `summary.md`. This also covers the case no CLI flag can: a tool *result* echoing a token back into the transcript. Three properties are load-bearing, each of them a defect found in review of the first implementation:
  - **Every spelling, not just the one the author typed.** Matching the raw value against serialized JSON misses the escaped form, and nearly every artifact here is JSON: `a"b` is stored `a\"b`, `ö` is stored `ö`. Structured artifacts are now scrubbed *before* serialization (no encoder in the path); text artifacts match the escaped forms too, since a CLI's own JSONL stream carries them. Not exotic for a credential — base64 contains `/` and `+`.
  - **Copied artifacts, not just written ones.** Redacting at the writers covers only what the harness authors. The agent's `workspace/` is *moved* into the artifact tree wholesale, so a token an MCP result echoed back and the agent saved to a file survived there. Scrubbed on bytes (binary-safe), after grading (never changing what was graded), names as well as contents, and not *through* symlinks.
  - **Run scope as well as cell scope.** `_secrets` is cleared per cell by design; the summaries are written after the last cell and *aggregate* cells, and `RunResult.error` carries a tail of child output — so they scrub against the run-scoped union.
- **The workspace scrub establishes completion or fails the cell** (second review round; the three properties above were each still true and each still incomplete). A traversal is only as good as what it can *see* and what it treats as an object:
  - **Write-and-rename, never in place.** A hardlink shares its inode with every other name pointing at it, so an in-place rewrite reached out of the artifact tree and overwrote an external file with `«redacted»`. The replacement is written beside the original and renamed over it: new inode for the archive, the agent's link target untouched. Refusing to follow *symlinks* never covered this — a hardlink has no target to refuse.
  - **A symlink's target string is archived even though its contents are not.** `readlink` reads it straight back, so a blandly named link pointing at `/tmp/<token>` published the credential as plainly as a file containing it. The target is redacted in place; traversal is still refused.
  - **`os.walk` reports an unreadable directory to `onerror` and then yields nothing for it** — indistinguishable from an empty one, so a `chmod 000` subtree was skipped in silence while the scrub reported success. Owner permissions are repaired before the walk (the harness owns this tree), and anything still unreadable is removed rather than published unchecked.
  - **Names are checked as assembled paths.** A secret containing a separator (`tenant/key`, a URL with credentials) spells itself out across a directory and its child while every individual component looks clean. The component that *completes* the secret is renamed; the prefix, not being a secret on its own, is left alone.
  - **The scrub returns what it could not certify**, and each such path is deleted and named in `RunResult.error`, failing the cell. A scrub that cannot prove a tree is clean must not report success over it — and a silent deletion would be worse than either outcome it is choosing between.
- **The scrub does not trust the shape of its own root, its entries, or its answers** (third review round; each of the five properties above survived, and each was still reachable around). The remaining leaks were all *identity* — what an object is, and whether the tree it lives in is the tree we think:
  - **`os.path.isdir` follows symlinks, so the root itself was a way in.** An agent that replaced its workspace with a link aimed the entire scrub at an external directory and had every file under it rewritten to `«redacted»`. The root is now classified with `lstat`; anything that is not a real directory is dropped whole and replaced with an empty one.
  - **The permission repair wrote through a hardlink.** Reading a `000`-mode file needs a `chmod`, and a `chmod` changes the *inode* — which is exactly what the previous round established a hardlink shares. Write-and-rename fixed the bytes and left the mode as an unguarded second channel to the same object. A multiply-linked file that cannot be read is now quarantined instead: dropping our name for it leaves every other name alone.
  - **"Not a directory" is not the same claim as "a readable regular file."** The dispatch treated every non-directory as one, so a FIFO made `open(...).read()` wait for a writer that never came and hung the run rather than failing it. Entries are classified with `lstat`; FIFOs, sockets and devices hold no archivable bytes and are removed and named.
  - **Not all of a file's bytes are in the file.** `xattr -w com.example.token <secret> note.txt` parks a credential in metadata that `cat` never prints, `cp -p`/`ditto`/`rsync -X`/zip all preserve, and a scrub reading contents reports `lost == []` over. Attribute names *and* values are scrubbed, on files, directories and symlinks (`xattrs.py`; CPython exposes the `os.*xattr` family on Linux only, so darwin goes through libc).
  - **A quarantine that cannot delete was reporting a deletion.** `chflags uchg` makes a file unremovable; the old code swallowed the `unlink` failure and returned the path as *lost*, which promises the artifact is gone. Removal now escalates outward-in — parent directory first (inside the tree, never hardlinkable elsewhere), then the entry's own flags — and the result is the answer to `lexists`, not the absence of an exception. What still cannot be removed raises `ScrubFailed`, whose sentence says the secret is **still on disk** rather than claiming a removal.
  - **A symlink is not an unshareable object.** `os.link(src, dst, follow_symlinks=False)` works on macOS, so an external symlink hardlinked into the workspace shares its inode with the archived name and an in-place `setxattr` rewrote the outside one's metadata. Symlinks are now replaced on a fresh inode like regular files (`_scrub_link`) — target and attributes rebuilt, renamed over — rather than edited.
  - **One pass of assembled-path reasoning can create the leak it just removed.** With `secretA` and `tailpart/childsecret` both declared, renaming `secretAtailpart` to clear the first spelled out the second across the new parent and a child the bottom-up walk had already gone by. Per-component scrubbing and assembled-path scrubbing are now separate phases, and the second runs **to a fixed point** — shallowest offender first, since renaming one component cleans every path beneath it — with a bounded round count that falls back to deletion rather than looping.
- **The scrub's reach is the archived tree, so the agent must never have a path into it** (fourth review round). Two of that round's three findings were still the scrub's; the third was upstream of it and larger: `exec_ws` — the directory the agent actually runs in — was relocated to a detached tempdir **only under `self.isolated`**. A `--no-isolated` cell therefore ran with its cwd at `cell_dir/workspace`, and `../agent-created.txt` published the raw secret in `cell_dir`, which `_scrub_tree` never examines because only `workspace` is archived. A cwd is a place the agent can *write*, not only one it can `list_dir`; `--no-isolated` means "see my real HOME and installed skills" and never meant "run inside the results directory". The relocation is now unconditional, and `exec_ws` is nested one level inside the harness's own tempdir so that `..` reaches a directory that is deleted whole rather than published. HOME isolation and execution-directory isolation answer different questions and no longer share a flag.
  - **A credential directory is deleted when it is gone, not when the call returns** (fifth review round). Both tempdirs that carry secrets — the agent's `exec_root` and the MCP `mcp_scratch`, whose `mcp.json` holds the interpolated `${VAR}` values in the clear — were removed with `shutil.rmtree(..., ignore_errors=True)`. An agent that `chmod 000`-ed its own execution directory left both secret-bearing files on disk, and the cell said nothing. Both now go through `_purge`, the same verified outward-in removal the workspace quarantine uses, whose result is `lexists`; a directory that genuinely cannot go **fails the cell** with the path named, before `_secrets` is cleared. `result.json` is re-written at that point too: it is serialized at step 6, long before the workspace is moved, so a note appended afterwards had been reaching `report.md` and nothing else.
  - **Absence and refusal are different answers.** `_scrub_tree` opened with `except OSError: return []`, so a cell directory at mode `000` produced a clean bill of health over a workspace still holding the secret. Only `ENOENT` is clean now; anything else repairs through the parent (`cell_dir`, which the harness created and owns) once, and raises `ScrubFailed` if that is not enough.
  - **A finding held in a frame the exception unwinds is a finding nobody reads** (sixth review round). The verified removal was right; where its answer *lived* was not. The scratch-dir failure was collected into a `purge_notes` local of `_run_cell_body`, and every path out of that function that is not a `return` discards it: `execute()` raising transfers straight to `_failed_cell`, which had no way to name a directory the body had created. Review made both happen at once and got a cell reporting `RuntimeError` and nothing about the `mcp.json` still on disk; the same loss followed any raise between the agent exiting and the notes being applied. Ownership now sits in `_CellCleanup`, held by `_run_cell` — the frame with the `try/except/finally`, the one an exception in the body cannot unwind. The body *registers* directories and purges each at the moment that suits it; `_failed_cell` sweeps whatever is left and drains the accumulated notes onto the crashed cell's own result; the outer `finally` warns about anything no result was written for. The guard also now starts where the directory does rather than at `execute`: resolving `${VAR}`s and building `RunOptions` sit inside the same `try`, because a raise from either used to leak the scratch dir and the isolated home outright.
    - **Reading a finding must not destroy it either** (seventh review round). The first version of that handoff drained on read, which put the note back in a local — one frame later than the bug it replaced, and just as lossy: review made the `result.json` rewrite raise between the read and the write, and the scratch failure vanished from a cell that then reported only the write error. `pending()` is now non-destructive and `acknowledge()` is called *after* the artifact writes return, so a note is forgotten when it reaches a reader rather than when it is handed to something that intends to carry it. The distinction is load-bearing one write further out too: acknowledging after `result.json` but before `report.md` looks safe because the note is on disk, right up until the crash path rewrites `result.json` from a rebuilt result that no longer has it. Whatever is still held when `_run_cell`'s finally runs is exactly what no artifact recorded, and gets echoed there.
    - **The isolated HOME is a resource on the same terms.** It was built at step 4 and removed by a guard that began at step 5, so anything raising in between — the overlay build, a progress update, the effort resolution — left `ase-home-*` behind and unnamed. It is now registered the moment `mkdtemp` returns, including on the build-failure paths, and its severity travels with the registration: `fatal=False`, because it holds config masks and symlinks into the real home. A stubborn one lands on `warnings` (durable via `cell.json`/`report.md`) instead of failing the cell, and its sentence says no credentials are in it rather than borrowing the credential wording.
- **Post-run warnings are recorded, not just printed** (`notices.py`). `execute()` archives the *child's* stderr; the harness process's own was written to a terminal and nowhere else, so a warning explaining an inexplicable result was itself unreadable afterwards — the claude server-health message said assertions would "fail for a reason the results will not show" while being that reason. Warnings now land on `RunResult.warnings` → `cell.json`, `report.md`, `summary.json`. The sink is **thread-local**: cells run in parallel and `sys.stderr` is process-global, so a `redirect_stderr` at the call site would file one cell's findings on another's result. Terminal rate-limiting (drift warnings, once per version) governs the *echo* only — every affected cell records it, or a matrix would show cell 1 carrying a finding that applies equally to cells 2..n.

## 6. Decision points

**A. Config delivery for claude/copilot: inline argv JSON vs scratch file vs workspace `.mcp.json`.**
Inline JSON leaks resolved secrets into `result.json` (argv is archived) and needs shell-safe quoting. Workspace files are archived and inlined into `report.md`; claude ignores workspace `.mcp.json` under `--strict-mcp-config` anyway. → **Scratch file** in the per-cell tempdir, deleted post-run.

**B. Codex delivery: `-c` overrides vs materialized `$CODEX_HOME/config.toml`.**
`-c` needs no isolation surgery, keeps HTTP secrets env-indirect natively, and its one exposure (stdio env literals in argv) is covered by redaction. A materialized config home means converting `.codex` from a wholesale symlink into a masked dir with a merged TOML (preserving auth) — heavier and riskier. → **`-c` overrides**, with config-home materialization as the fallback if TOML array/inline-table `-c` values prove unreliable (verification task).

**C1. Allowlist shape: flat top-level `allowed_tools: [mcp__sliderule__*]` vs per-server `tools:`.**
Flat matches claude's flag but forces claude's naming onto codex/copilot (whose native filters take bare tool names per server) and repeats server names. → **Per-server `tools:`**; adapters compile to their native form. A flat form can be added later without breakage.

**C2. Claude gating strength: advisory vs hard.** — **resolved by measurement (2026-07-22, claude 2.1.113), and the answer inverted the ranking.** All three options were run against the `fixtures/echo_mcp_server.py` stdio server (tools `echo` + `add`), allowing `echo` and instructing the model to call `add`:

| option | result |
|---|---|
| (a) `--dangerously-skip-permissions` + `--allowedTools mcp__echo__echo` | **No gating whatsoever.** `mcp__echo__add` stayed in the init `tools` list, was called, returned `42`. Advisory is not even advisory here — nothing carries the preference to the model. |
| (b) `--permission-mode dontAsk` + `--allowedTools mcp__echo__echo` | **Hard, at call time.** The tool is still advertised, the model still attempts it, and the *call* is refused (`is_error: true`, "running in don't ask mode"). But the allowlist governs **every** tool: with only the MCP tool listed, `Write` and `Bash` were both denied and the file never written. Enumerating the whole built-in surface is version-brittle, and it changes the agent's capabilities — which makes a gated cell non-comparable with an ungated one, the exact axis `Runner._consistency` exists to protect. |
| (c) `--dangerously-skip-permissions` + `--disallowedTools mcp__echo__add` | **Hard, at advertisement time** — strictly the cleanest. The denied tool is **absent from the init `tools` list**; the model never learns it exists and reports the server "only exposes an `echo` tool". Built-ins are untouched. |

Two further measured facts shape the mapping: `--disallowedTools` accepts wildcards (`mcp__echo__*`), and **deny beats allow** — `--disallowedTools "mcp__echo__*" --allowedTools "mcp__echo__echo"` removed *both* tools. So the tempting "deny the namespace, re-admit the allowlist" is not expressible. And `--mcp-config` has no native per-server filter: `tools`, `allowedTools`, and `enabledTools` in a server object are all **silently ignored** (accepted, not rejected — so a plausible-looking key in a hand-written config does nothing quietly, which is worth a validation error on the harness's own schema).

→ **(c)**, with the deny set computed as the *complement* of `tools:` — and with the strength of that guarantee stated in two parts, because the original objection to (c) is **not** fully closed. An earlier revision of this section said it was "closed the same way every other Phase 0 gap was". That cited the precedent while dropping the qualifier that made the precedent honest: those gaps were *classified* as detection, not closed (§8's copilot audit says so in as many words), and the same classification is owed here.

Computing a complement requires the server's full tool list, which is not statically available at argv-build time. The harness gets it by starting the server and calling `tools/list` itself — a **second, independent server instance** from the one claude launches. That is precisely the two-execution gap the Windows ODR registry gate and copilot's `git rev-parse` window are refused on: a stateful or time-varying server can advertise `[echo]` to the enumerator and `[echo, add]` to claude, and `add` — never having entered the complement — is advertised and callable. The post-run init check *detects* exactly that, and detects it exactly (denied tools are absent from init's `tools` list, measured above; claude advertises its full surface unconditionally, so there is no idle-object equivalent of codex's ABA hole). But it detects it after the call may already have had effects, under `--dangerously-skip-permissions`. Full init visibility closes an evidence gap, not an enforcement window.

So:
- **Prevention** for the tool set observed at enumeration — hard, advertisement-time removal, not advice.
- **Detection, fail-closed**, for anything appearing afterward: any advertised `mcp__<server>__*` outside the declared allowlist fails the cell.

The harness therefore cannot promise an off-list tool was never *callable*. It promises that no run **reported clean** had an off-list tool advertised to its model — deliberately the same shape, and the same words, as copilot's `verify_post_run` guarantee (§4). Two consequences worth naming: enumeration **starts the declared server an extra time**, so a server with startup side effects performs them twice and a remote one is contacted twice with the scenario's credentials; and the gap exists only for scenarios that actually use `tools:` — omit it and there is no complement, no enumeration, and no second instance.

Hard prevention against a server the scenario author does not control needs the filtering proxy in **C3**. `tools:` on claude is not a security boundary against a hostile server, and neither the schema docs nor the error messages may imply it is.

**C3. A harness-owned filtering proxy — deferred, but the only mechanism that is both hard and uniform.**
Rather than compiling `tools:` to four different native filters, the harness could launch its own stdio shim as the `command` the CLI actually starts; the shim spawns the declared server, filters `tools/list` responses down to the allowlist, and refuses off-list `tools/call` at the wire. That closes C2's drift window outright — there is exactly **one** server instance and the harness sits inside the connection, so there is no second execution to disagree with — and it is the only thing that would give **antigravity** tool gating at all (§2: no native filter exists there; today the schema can only warn). It would also produce wire-level MCP telemetry that no CLI reports uniformly, which is the same reason `--output-format json` became load-bearing for copilot.

Deferred out of Phase 1 for two reasons: it puts harness-authored code in the request path of every gated run, where a bug is a silently wrong eval rather than a loud failure; and remote servers need it to be a transport *bridge* (stdio in, HTTP/SSE out), which is materially more than a pipe pump. → **Phase 1 ships C2's native mechanism with C2's stated guarantee.** Revisit before any scenario points `tools:` at a server its author does not control, and revisit for agy regardless — that adapter has no other path to tool gating.

**D. Where secrets interpolate: harness at load time vs CLI-native expansion.**
CLI-native keeps values off disk but is inconsistent (copilot/agy have nothing). → **Hybrid**: harness interpolation as the uniform contract + fail-fast validation; adapters use native indirection where available (codex); redaction pass regardless.

- **Phase 1 — schema + claude** (**implemented**; codex moves to Phase 1b): `mcp_servers:` parse/validate/interpolate (`mcp.py`), the secrets registry + redaction pass, per-cell `ase-mcp-` scratch dir, claude's `--mcp-config` file, and `scenarios/mcp_echo_smoke.yaml` over the echo fixture.   Four decisions worth carrying forward, all of them refusals:
  - **`tools:` on claude is a validation error**, not an accepted no-op — the enumeration step of §6-C2 is not built, and an allowlist that is accepted without being enforced is worse than none, because nothing in the results would show it never applied. The message names C3.
  - **codex/copilot/antigravity refuse `mcp_servers:` outright** rather than running without them. Silently dropping declared servers would grade a scenario that never had the tools it asked for, and every MCP assertion would fail for a reason that looks nothing like the cause.
  - **`${VAR}` is honoured in `env`/`headers`/`url` only, never in `command`/`args`.** Interpolating into the executable would turn an environment variable into a way to choose what program runs — a categorically larger power than supplying a credential.
  - **The CLI-native filter spellings are rejected by name** (§3), with the error naming `tools:`, because claude accepts and silently ignores them.

  `_mcp_witness` now permits the *declared* set and only that — previously any named server was a kill-switch violation, so every intentional server failed the run closed. A declared server that does **not** appear warns rather than raises: nothing leaked, but the scenario ran without the surface it asked for. So does one that appears in any state other than `connected` — being **named** in the witness is not the same as being **usable**, and `{"name": "echo", "status": "failed"}` used to clear the check in silence (neither undeclared nor missing). An unrecognised status warns too: a state this adapter does not understand is not evidence of health. An *undeclared* server still fails the run whatever its status claims — `status` is input from the server's own host, so it may qualify a declared server's health but never excuse an unexpected one's presence.

  A second review round found the four refusals above lived **only in the CLI pre-flight**, so `Runner.run()` or any direct caller could route around every one of them. They are re-asserted at `exec.execute`'s `build_argv` call — the one choke point every invocation passes through — inside the block whose failure mode is already "could not construct a hermetic invocation", so an invocation that cannot be built to the scenario's spec becomes a failed run rather than a quietly weaker one. Validation that only guards the CLI is a lint, not an invariant.

  Known residual, deliberately not papered over: `mcp_servers_seen()` returns `None` once `--mcp-config` is on argv, so an MCP-enabled matrix reads `unverified` on the MCP axis. That is the honest answer — argv names a file, not a server set — and it fails safe under the every-axis rule rather than manufacturing a green. Reporting the *witnessed* set from the init event would be strictly better and is the follow-up.
- **Phase 1b — codex**: `-c` mapping + canonical `mcp__server__tool` naming in its parser. Blocked on §9 probe #2 (TOML array/inline-table values via `-c`), and best done together with per-cell `$CODEX_HOME` materialization — that closes codex's ABA hole and would make it the first adapter able to set `parallel_safe_config`.
- **Phase 2 — copilot**: `--additional-mcp-config @file`, per-server `tools`, `--secret-env-vars`, verify MCP tool naming in its JSON events, parser tweak if needed.
- **Phase 3 — antigravity**: config materialization on the Phase-0 mask mechanism, `serverUrl`/`url` support, tool-name normalization; document the gaps (no tool gating; isolation required).
